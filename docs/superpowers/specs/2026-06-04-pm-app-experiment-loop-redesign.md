# PM App Experiment Loop Redesign

**Date:** 2026-06-04
**Scope:** Webhook fix, trigger config rework, sprint feedback UI, bundle prompt versioning, VIDF git hook

---

## Core Objective

The PM App runs a controlled experiment: does giving developers a structured resource bundle (generated by Claude) before they start a feature improve code quality over time? The system tracks this by attaching a bundle version tag to every git commit and using git blame to compare outcomes across bundle versions.

---

- Next.js App Router on Vercel
- Supabase schema: tasks, lists, users, oauth_tokens, trigger_queue, trigger_configs, sprints, objective_assessments, skills_library, slack_issues, bot_sops, bot_observations, sop_proposals, bundle_generations

- Setup page, Sprint Planner, FVI Assessment pipeline (7-phase), Bundle generation (8-file commit to docs repo), Assessment history
- Full Slack support bot (issue triage, SOP system, observation layer)

## Workflow (source of truth)

Tasks live in four ClickUp lists. The PM moves tasks manually. The PM App reacts to `taskMoved` webhook events only.

| List | On Move | PM App Action |
|---|---|---|
| Planning | PM moves task here | No automated action |
| Active | PM moves task from Planning | Cherry-pick bundle commit → `active/{id}-{slug}` branch; post kickoff comment on ClickUp task |
| Next Release | PM moves task from Active | Archive `active/{id}-{slug}` branch (merge to docs main); strip CLAUDE.md from feature worktree |
| Archive | PM moves task | Close vault branch; mark spec archived |

## Workstream 1: Webhook Handler Fix

### DB Migration: `014_trigger_configs_list_routing.sql`
- Add `destination_list_id UUID REFERENCES lists(id)` to `trigger_configs`
- Make `to_status` nullable (was `NOT NULL`)
- Add index on `destination_list_id`

### `lib/clickup/webhook.ts`
No change needed. `parseWebhookEvent` already extracts `listId` on `taskMoved` events.

### `app/api/webhooks/clickup/route.ts`
**Current bug:** handler queries configs by `(list_id, to_status)` using the task's current list and a status string. On `taskMoved`, `toStatus` is `''`. Task's `list_id` in DB is never updated on move.

**Fix:**
- On `taskMoved`: query `trigger_configs` where `destination_list_id = event.listId`. Update task's `list_id` to the destination list. **Do not update `status`** — `status` is authoritative from `taskStatusUpdated` events only; leaving it stale-but-correct is preferable to blanking it (Sprint Planner and FVI pipeline both filter on `status`).
- On `taskStatusUpdated`: keep existing behavior.
- Auto-import path: when task not tracked, set `list_id` from destination list lookup. Leave `status` as `null` (not `''`).
- `handleSlackHandoff` is correct as-is — no change.

**v1 limitation:** The experiment tag does not include a per-task ID. Developers working two tasks simultaneously in the same sprint will have commits tagged with the same sprint/bundle context. Per-task linkage is deferred to a future experiment version; v1 analysis correlates commits to tasks via ClickUp assignment data post-hoc.

## Workstream 2: Seed Trigger Configs

### New env vars required
Three env vars must be added to `.env.local` and Vercel before running the seed:
- `CLICKUP_ACTIVE_LIST_ID`
- `CLICKUP_NEXT_RELEASE_LIST_ID`
- `CLICKUP_ARCHIVE_LIST_ID`

(`CLICKUP_PLANNING_LIST_ID` already exists in the codebase.)

### `scripts/seed-trigger-configs.ts`
One-shot script (`npx tsx scripts/seed-trigger-configs.ts`):
1. Reads 4 list IDs from the env vars above
2. Looks up corresponding `lists.id` UUIDs in Supabase
3. Upserts 4 `trigger_configs` rows keyed on `destination_list_id`

| Destination List | `pm_agent_action` |
|---|---|
| Planning | `noop` |
| Active | `cherry_pick_bundle_and_post_kickoff` |
| Next Release | `archive_active_branch` |
| Archive | `close_vault_branch` |

`pm_agent_action` values are the enum strings the trigger queue processor matches on.

**Idempotency guards (trigger queue processor, not the webhook):**
- Before `cherry_pick_bundle_and_post_kickoff`: check that `active/{id}-{slug}` branch does not already exist in the docs repo. If it does, skip and mark queue item `done` with a note.
- Before `archive_active_branch`: check that the `active/{id}-{slug}` branch is still open (not already merged). If already merged, skip and mark `done`.
These guards make it safe for a PM to accidentally drag a task back and forth without creating duplicate branches or git errors.

## Workstream 4: DB Migrations

### `016_bundle_feedback.sql`
```sql
CREATE TABLE bundle_feedback (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id UUID NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  sprint_id UUID REFERENCES sprints(id) ON DELETE SET NULL,
  bundle_version INT NOT NULL,
  developer_email TEXT NOT NULL,
  kickoff_prompt_rating INT NOT NULL CHECK (kickoff_prompt_rating BETWEEN 1 AND 5),
  user_stories_rating INT NOT NULL CHECK (user_stories_rating BETWEEN 1 AND 5),
  dev_skill_rating INT NOT NULL CHECK (dev_skill_rating BETWEEN 1 AND 5),
  comments TEXT,
  submitted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(task_id, developer_email)
);
```

### `017_bundle_prompt_versions.sql`
```sql
CREATE TABLE bundle_prompt_versions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  version INT NOT NULL UNIQUE,
  prompt_text TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'archived')),
  proposed_prompt_text TEXT,
  change_summary TEXT,
  activated_at TIMESTAMPTZ,
  approved_by TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Also adds `prompt_version INT` column to existing `bundle_generations` table so each generated bundle is stamped with the prompt version that produced it.

One row has `status = 'active'` at any time. Bundle generation always fetches the active version.

## Workstream 5: Sprint Feedback UI

### Route: `/feedback?token={signed-token}` (no auth, token-gated)

The PM shares a signed URL at sprint close (via ClickUp comment or Slack). The token is a Node `crypto` HMAC containing `sprint_id` and `expires_at` (Unix timestamp, 7 days from generation), signed with `FEEDBACK_TOKEN_SECRET`. Validation: verify HMAC signature, then check `Date.now() < expires_at`. On failure: show "Invalid or expired link" — no task data returned.

**Page flow:**
1. Developer follows the PM-shared URL — token is validated server-side, sprint ID extracted. If token is missing or expired, page shows an "Invalid or expired link" message.
2. Developer enters their email (identifier only — no password).
3. Per-task feedback card:
   - Task name + bundle version used
   - Three 1–5 star ratings: Kickoff Prompt, User Stories, Dev Skill
   - Optional free-text comments
4. Submit → POST `/api/feedback/bundle` → upserts into `bundle_feedback` (keyed on `task_id + developer_email`).
5. Thank-you state after submit.

### `POST /api/feedback/bundle`
Payload: `{ token, email, responses: [{ task_id, bundle_version, ratings, comments }] }`. Validates token, validates ratings 1–5. Upserts. Token validation replaces auth — IP rate limiting also applied.

## Workstream 6: Bundle Prompt Versioning Flow

### Route: `/experiments/prompt-versions` (auth required)

Two-panel layout:

**Left — Feedback Summary**
Aggregate ratings from `bundle_feedback` grouped by `bundle_version`. For the active version: average scores per dimension, total responses, scrollable free-text comments. "Analyze & Propose Changes" button appears when ≥1 unreviewed responses exist.

**Right — Proposed Diff**
Empty until PM clicks "Analyze & Propose Changes".

### `POST /api/experiments/propose-prompt`
1. Fetch all `bundle_feedback` for active version
2. Fetch active `bundle_prompt_versions` row
3. Send to Claude: analyze feedback, return the **complete rewritten prompt text** + a bullet-point summary of what changed and why (no diff format — full text replacement avoids patch application errors)
4. Store result in `proposed_prompt_text` and `change_summary` on the active version row
5. Return both fields to UI

Right panel shows `change_summary` above a scrollable `proposed_prompt_text` block. Two buttons: **Approve** / **Reject**.

### `POST /api/experiments/approve-prompt`
Single transaction:
1. Set current active version `status = 'archived'`, clear `proposed_prompt_text` and `change_summary`
2. Insert new row: `prompt_text = proposed_prompt_text`, `status = 'active'`, `version = prev + 1`, `activated_at = now()`, `approved_by = session.user.email`

**Reject:** Clear `proposed_prompt_text` and `change_summary` on current row, reset right panel.

## Workstream 7: VIDF Git Hook + GitHub Action

### `scripts/vidf-hook/install-git-hook.sh`
Run once in a feature repo. Writes `commit-msg` hook to `.git/hooks/commit-msg`:
- Reads `git config user.email`
- Calls `GET /api/developers/{email}/experiment`
- Appends `[vidf:{version} | bundle:v{N} | sprint:{YYYY-MM}]` as a new line in the commit message body if not already present
- **Falls back gracefully (exit 0, no tag appended) on any failure** — API unreachable, non-200 response, missing/malformed JSON. The GitHub Action is the enforcement point; the hook must never block a commit.

### `GET /api/developers/[email]/experiment`
No auth. Returns:
```json
{ "version": "v1", "bundle_version": 3, "sprint": "2026-06" }
```
Derives from: active `bundle_prompt_versions` row, current open sprint (`sprints` table), hardcoded experiment version `v1`.

### `.github/workflows/vidf-validate.yml`
Runs on `pull_request`. Commit enumeration:
```bash
git log origin/main..HEAD --no-merges --oneline
```
This isolates only commits introduced in the feature branch, excluding merge commits from upstream. Validates each commit message against `[vidf:* | bundle:v* | sprint:*]`. Fails with a message listing offending commit SHAs. Only activates when `.vidf-enabled` file exists in repo root — opt-in to avoid breaking other Viscap repos.

## Implementation Order

1. **Webhook fix + Seed trigger configs — deploy together (atomic).** The webhook fix must not go live before valid `trigger_configs` rows exist. Deploy migration 014 + the seed script output + the updated route handler as a single unit. An empty config layer would silently discard incoming ClickUp events.
2. Trigger Config UI (PM-facing correctness — no deploy dependency)
3. DB migrations — `bundle_feedback` + `bundle_prompt_versions` (foundation for 4 and 5)
4. Sprint feedback UI (depends on migration in step 3)
5. Bundle prompt versioning flow (depends on migration in step 3)
6. VIDF git hook + GitHub Action

Each workstream ships as an independent PR except 1+2 (webhook + seed), which ship together.
